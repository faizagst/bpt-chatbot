Ringkasan Eksekutif

Gambaran tujuan, permasalahan, dan nilai strategis sistem

1.1 Tujuan Sistem

LENTERA (Layanan Talenta Digital Responsif AI) adalah sistem chatbot berbasis kecerdasan buatan yang dikembangkan untuk Balai Pelatihan Talenta Komunikasi dan Digital (BPT Komdigi), Kementerian Komunikasi dan Digital Republik Indonesia. Sistem ini dirancang untuk memberikan layanan informasi yang akurat, konsisten, dan tersedia 24 jam kepada masyarakat umum mengenai program-program pelatihan yang diselenggarakan oleh BPT Komdigi.

LENTERA menggunakan pendekatan Retrieval-Augmented Generation (RAG) — teknologi AI generatif yang diperkaya dengan basis pengetahuan dari dokumen resmi BPT Komdigi, sehingga setiap jawaban yang diberikan bersumber dari informasi yang terverifikasi, bukan dari pengetahuan umum model AI semata.

1.2 Permasalahan yang Diselesaikan

1.3 Manfaat Utama

🎯 Akurasi Tinggi
Jawaban bersumber dari dokumen resmi BPT Komdigi, bukan dari "halusinasi" AI umum. Sistem secara eksplisit menyatakan ketika informasi tidak ditemukan.
⚡ Respons Cepat
Cache berlapis (exact + semantic) memastikan pertanyaan serupa dijawab dalam milidetik tanpa memanggil Gemini AI lagi. Efisiensi biaya API tinggi.

Admin mendapat insight real-time: pertanyaan apa yang paling sering ditanyakan, clustering semantik mengelompokkan pertanyaan bermakna sama.

2.2 Alur Data RAG — Saat User Bertanya

1

User mengetik pertanyaan di ChatWidget

Browser mengirim POST /api/rag/chat ke Next.js frontend (session_id disimpan di localStorage)

2

Next.js Proxy meneruskan ke FastAPI Backend

Route /api/rag/chat di Next.js mem-proxy ke http://backend:8000/api/chat (Docker internal)

3

Rate Limit Check (Redis)

Backend memeriksa apakah session sudah mencapai batas 10 pertanyaan/3 jam. Jika terlampaui, langsung return HTTP 429 dengan TTL countdown.

4

Exact Cache Lookup (Redis)

Backend mencari cache hash SHA-256 dari query + data_version + parameter model. Cache hit → jawaban langsung dikembalikan tanpa LLM call.

5

Semantic Cache Lookup (Qdrant)

Jika exact cache miss, query diubah menjadi embedding lalu dicari di Qdrant. Pertanyaan yang "bermakna sama" (skor cosine similarity ≥ 0.92) akan mengambil jawaban dari cache ini.

6

Retrieval — Cari Dokumen Relevan (Qdrant)

Query di-embed ke vector lalu dicari di koleksi dokumen Qdrant. Top-K (default: 3) chunk dokumen yang paling relevan diambil sebagai konteks.

7

Generate Jawaban (Gemini 2.5 Flash)

Konteks dokumen + pertanyaan user dikirim ke Gemini melalui template prompt. Gemini diminta menjawab HANYA berdasarkan konteks, dan wajib menggunakan kalimat fallback standar jika tidak ada info.

8

Simpan & Cache Hasil

Jawaban disimpan ke: MongoDB (chat log), Redis (exact cache), Qdrant (semantic cache). Chat log cache top-questions diinvalidasi agar data analytics tetap fresh.

9

Response ke Frontend

Frontend menerima: answer, sources (nama file dokumen), quota (sisa pertanyaan), is_fallback (boolean apakah info tidak ditemukan).

2.3 Alur Autentikasi Admin

1

Admin mengakses /admin → tampil LoginPage

Next.js middleware melindungi /admin/dashboard — user tanpa JWT cookie diredirect ke /admin

2

POST /api/auth/login — verifikasi password

Server membandingkan password dengan bcrypt hash di MongoDB. Jika cocok, JWT token dibuat (24 jam), disimpan sebagai HttpOnly cookie.

3

Redirect ke /admin/dashboard

Middleware Next.js memverifikasi JWT pada setiap request ke /admin/dashboard/* menggunakan library jose.

4

Session timeout otomatis 24 jam

AdminPanel memiliki countdown timer. Saat JWT expired, cookie dihapus otomatis dan admin diredirect ke halaman login kembali.

Fitur Sistem Detail

Penjelasan teknis mendalam setiap fitur utama

5.1 RAG Pipeline & Knowledge Base

Proses Upload Dokumen

Ketika admin mengupload dokumen, backend melakukan pipeline berikut:

  1. Deteksi format file berdasarkan ekstensi
  2. Ekstrak teks: PDF → pdfplumber, DOCX/DOC → docx2txt, CSV/XLSX → pandas, PNG/JPG/JPEG → EasyOCR (deep learning OCR), TXT → langsung dibaca
  3. Chunking: teks dipecah menjadi chunks ukuran 1000 karakter dengan overlap 100 karakter menggunakan RecursiveCharacterTextSplitter
  4. Embedding: setiap chunk di-embed menjadi vector 768 dimensi menggunakan text-embedding-005
  5. Store ke Qdrant: setiap vector disimpan dengan metadata doc_id dan filename
  6. Store ke MongoDB: metadata dokumen (doc_id, filename, uploader, status, uploaded_at) dicatat
  7. Invalidate cache: data version di Redis di-bump agar jawaban cache lama tidak dipakai lagi

Prompt Engineering

Sistem menggunakan prompt yang secara eksplisit membatasi LLM agar menjawab HANYA dari dokumen yang tersedia:

5.2 Sistem Cache Berlapis

Layer CacheStorageMekanismeTTLKeterangan
Exact CacheRedisHash SHA-256 dari: query (normalized) + data_version + semua parameter model1 jamHit jika pertanyaan benar-benar sama persis setelah normalisasi
Semantic CacheQdrant + RedisCosine similarity embedding query vs cache index (min score 0.92, min token coverage 0.65)1 jamHit jika pertanyaan "bermakna sama" walau kata berbeda
Top-Questions CacheRedisHasil komputasi clustering semantic disimpan setelah selesai10 menitInvalidated saat ada chat baru atau clear history

Data Versioning: Setiap kali dokumen diupload atau dihapus, data_version di Redis di-increment. Cache mengandung data version saat dibuat, sehingga jawaban dari dokumen lama otomatis tidak dipakai setelah dokumen di-update.

5.3 Rate Limiting

Untuk mencegah penggunaan berlebihan dan mengontrol biaya API, setiap sesi pengguna dibatasi:

  • Batas pertanyaan10 pertanyaan per sesi
  • Window waktu3 jam (10.800 detik)
  • MekanismeRedis INCR + EXPIRE per session_id
  • Response saat limitHTTP 429 dengan ttl_seconds (countdown) dan info quota
  • Session IDUnique per browser (disimpan di localStorage, format session_{timestamp}_{random})
  • UIFrontend menampilkan countdown timer realtime (jam:menit:detik) hingga kuota reset
  • Fallback UIKetika kuota habis atau info tidak ada, ditampilkan card dengan link WhatsApp dan Ticketing System

5.4 Panel Admin

Panel admin tersedia di /admin/dashboard dan dilindungi JWT. Fitur yang tersedia:

📤 Upload Dokumen
Drag & drop atau klik pilih file. Format: PDF, DOC, DOCX, CSV, XLSX, PNG, JPG, TXT. Maks 15MB. Progress dan hasil upload ditampilkan realtime.
📋 Kelola Dokumen
Daftar semua dokumen yang diindeks di Qdrant. Info: filename, format, waktu upload, uploader, status. Tombol hapus dengan konfirmasi.
📊 Analitik Pertanyaan
Dashboard top-questions dengan semantic clustering. Menampilkan pertanyaan yang sering ditanya, jumlah varian, dan cluster pertanyaan bermakna sama.
💬 Riwayat Pesan
Log semua percakapan yang masuk. Filter pencarian per keyword. Timestamp dalam WIB. Tombol hapus semua riwayat.

Sesi admin otomatis expire setelah 24 jam, dengan countdown timer yang berubah warna merah saat mendekati waktu habis (kurang dari 5 menit).

5.5 Analitik & Monitoring

Semantic Clustering Top-Questions

Fitur analitik menggunakan model paraphrase-multilingual-MiniLM-L12-v2 yang berjalan lokal (tanpa biaya API) untuk:

  1. Mengambil semua pertanyaan dari MongoDB chat logs
  2. Mengubah setiap pertanyaan menjadi embedding vector 384 dimensi
  3. Mengelompokkan pertanyaan yang similarity-nya di atas threshold menggunakan hierarchical clustering
  4. Menentukan "canonical question" (pertanyaan representatif) untuk setiap cluster
  5. Mengembalikan cluster diurutkan dari yang paling banyak muncul

Hasil clustering di-cache di Redis selama 10 menit untuk performa, dan diinvalidasi saat ada percakapan baru.

Keamanan Sistem

Lapisan keamanan yang diimplementasikan

7.1 Autentikasi & Otorisasi

  • Password Hashingbcryptjs dengan salt rounds 10. Password plaintext tidak pernah disimpan.
  • JWT TokenSigned dengan secret key acak 48-byte. Berlaku 24 jam. Disimpan sebagai HttpOnly cookie (tidak bisa diakses JavaScript).
  • Middleware ProteksiNext.js Edge middleware melindungi semua route /admin/dashboard/*. Request tanpa JWT valid langsung diredirect ke login.
  • Library JWTjose — kompatibel dengan Next.js Edge Runtime, lebih aman dari jsonwebtoken

7.2 Keamanan Database

  • Isolasi JaringanMongoDB, Qdrant, dan Redis TIDAK di-expose ke port host. Hanya bisa diakses dari container lain dalam bpt_network.
  • Autentikasi MongoDBDatabase terlindungi dengan root username/password. Bukan mode no-auth.
  • Autentikasi RedisRedis dikonfigurasi dengan requirepass — semua koneksi wajib autentikasi.
  • GCP CredentialsFile gcp-key.json di-mount sebagai read-only (:ro) dan tidak pernah dimasukkan ke dalam image Docker.

7.3 Keamanan API

  • CORSDi production, header CORS hanya mengizinkan domain frontend yang dikonfigurasi via ALLOWED_ORIGINS. Bukan *.
  • Rate LimitingRedis-based per session ID. Mencegah abuse dan menjaga biaya API Vertex AI < terkontrol.
  • Input ValidationQuery chat dibatasi maksimal 100 karakter (Pydantic max_length). File upload dibatasi format dan ukuran.
  • HTTPS OnlyNginx meredirect semua HTTP ke HTTPS. Semua komunikasi terenkripsi TLS 1.2/1.3.
  • Secret ManagementSemua password dan API key disimpan di file .env yang tidak pernah di-commit ke repository Git (.gitignore).

Tips Kualitas Dokumen

  • PDF dengan teks selectable lebih baik daripada PDF hasil scan (gambar). pdfplumber dapat mengekstrak tabel dengan lebih akurat dibanding PDF hasil scan.

  • Dokumen dalam Bahasa Indonesia mendukung penuh, termasuk karakter khusus seperti é, ä, dll.

Keamanan File Environment

  • Semua file .env dan gcp-key.json TIDAK boleh dimasukkan ke Git repository
  • File-file ini wajib diupload secara manual ke VPS melalui SCP atau SFTP
  • Password MongoDB dan Redis harus berbeda satu sama lain dan minimal 16 karakter
  • JWT Secret harus berupa string acak minimal 48 byte (generate dengan PowerShell atau OpenSSL)
